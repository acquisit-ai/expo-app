# 项目整体需求、应用形态与技术栈分析框架

## 1. 文档目的

这份文档用于回答三个最核心的问题：

- 这个项目整体想做什么
- 这个 App 在用户视角下到底是什么形态
- 一个重构项目适合采用什么样的技术栈和工程组织方式

与偏“架构规范”和“文件树说明”的文档不同，本文档更强调**产品意图、应用整体描述和技术栈分层方式**。它适合在以下场景优先阅读：

- 第一次接手一个重构项目，想快速建立整体认知
- 需要撰写项目总览、产品简介或重构背景说明
- 需要把“业务需求、App 形态、技术栈”分层说明清楚
- 需要给其他项目复用一份结构化分析模板

## 2. 一句话描述

对于内容型学习应用，一个常见且有代表性的形态是：以**短视频 + 字幕学习**为主线，以**词汇沉淀与学习记录**为辅线。它不是传统意义上只做“背单词”的工具，而是更接近“通过可刷的视频内容驱动学习，再把学习结果沉淀到个人学习空间”的产品。

## 3. 项目整体需求

对于这类需要重构或重新梳理的应用，整体需求通常可以拆成四个层面。

### 3.1 内容消费需求

用户通常需要先有一个低门槛、连续性的内容入口，例如视频 Feed、课程流、音频流或内容推荐页。

这类需求包括：

- 浏览学习视频流
- 点击视频进入更深入的观看页面
- 在详情和全屏模式之间切换
- 在连续观看中保持流畅的加载、预加载和滚动体验
- 支持后台播放、画中画、横竖屏切换等接近内容平台的基础能力

这说明产品不是把“知识卡片”或“工具页”当成唯一入口，而是希望把学习嵌入到更自然、更沉浸的内容消费路径中。

### 3.2 学习辅助需求

仅有内容播放并不能支撑“学习型产品”的定位，因此通常还需要在内容之上叠加学习辅助层。

这类需求包括：

- 字幕加载与展示
- 字幕导航和定位
- 单词、短语、概念或语义片段的解释能力
- 播放速度、音量、进度控制
- 全屏观看与操作手势
- 与内容理解相关的交互反馈

换句话说，内容在这里不是消费终点，而是学习上下文的承载体。

### 3.3 学习沉淀需求

用户消费过内容之后，需要能够把学习结果沉淀下来，否则产品只会停留在“看视频”的层面。

这类需求包括：

- 把关键知识点、单词、短语或条目沉淀到个人空间
- 查看沉淀内容的详情
- 对沉淀内容做排序、筛选和分组
- 记录学习进度、收藏状态或掌握程度
- 从个人空间回看收藏、历史或复习清单

这一层需求把“短视频学习”延伸成“可积累、可回顾、可复习”的长期学习过程。

### 3.4 账户与个性化需求

一旦产品包含个人沉淀空间、收藏、历史和偏好设置，就需要完整的用户身份体系和个性化能力。

这类需求包括：

- 邮箱登录、验证码登录、忘记密码、密码设置
- 会话恢复和退出登录
- 主题切换
- 用户资料展示
- 个人收藏与历史记录

这一层说明项目目标不是一个匿名 Demo，而是准备朝“长期使用的个人学习产品”方向发展。

## 4. App 的整体描述

## 4.1 从用户视角看，App 可以如何描述

如果从用户进入产品后的实际体验来描述，这类 App 通常可以理解为：

一个带登录体系的学习型内容 App。用户登录之后，会先进入一个以内容流为核心的首页，然后可以：

- 浏览学习内容
- 进入详情或沉浸式观看/阅读/收听
- 借助字幕、注释或辅助信息理解内容
- 将有价值的知识点沉淀到个人空间
- 在个人页查看收藏、历史和偏好设置

它的重心更偏“内容驱动学习”，而不是“从工具页出发学习”。沉淀区是学习成果的归档空间，而不是唯一的业务主入口。

## 4.2 典型的 App 主要结构

从整体产品结构看，这类 App 的主应用区通常可以概括成三大块：

### 主入口：Feed

Feed 或内容流通常是应用的核心入口，也是最重的业务主线。

它承载了以下职责：

- 内容流
- 点击进入详情
- 沉浸式切换
- 连续消费体验
- 返回后的滚动位置恢复
- 预加载和窗口管理

在大多数内容型学习应用里，内容流和播放器/阅读器系统通常都是第一优先级。

### 学习沉淀区：Collections / Library / Notebook

这一块通常对应个人词汇本、知识库、收藏夹或学习沉淀空间。

用户可以在这里：

- 查看已沉淀的条目列表
- 进行排序
- 查看条目详情
- 看到同步时间、学习进度等辅助信息

这一块的价值在于把一次性消费转成长期可回顾、可组织、可复习的资产。

### 个人空间：Profile

Profile 负责承载用户自己的空间和系统级入口。

目前包含的方向有：

- 用户信息展示
- 主题切换
- 收藏入口
- 历史入口
- 设置、反馈、关于等扩展入口

这些入口共同组成“内容消费 -> 学习沉淀 -> 个人空间”的产品闭环。

## 5. 技术栈

## 5.1 客户端基础栈

对于这类移动端重构项目，常见的客户端基础栈是：

- React
- React Native
- Expo
- TypeScript

这是一套标准的现代移动端前端方案，适合在单仓库中统一管理 iOS / Android 应用逻辑。

## 5.2 导航与页面组织

导航相关栈包括：

- React Navigation
- Stack Navigator
- Bottom Tabs

这套导航体系支撑了：

- 认证区与主应用区分离
- Tab 主入口
- 视频详情与全屏的独立栈
- 收藏/历史等附加页面

## 5.3 状态管理与数据请求

常见组合包括：

- Zustand：负责客户端状态和领域状态
- TanStack Query：负责异步请求、缓存和刷新

整体思路比较明确：

- 实体状态放在领域 store 中
- 远程或异步数据通过请求层进入状态层
- 页面只消费统一状态和 Feature 能力

这说明工程设计偏向清晰分层，而不是把所有逻辑塞进页面组件。

## 5.4 认证与会话

认证相关能力通常依赖：

- Supabase Auth
- 本地持久化存储

它通常承载：

- 邮箱密码登录
- 验证码登录
- 会话恢复
- 自动刷新
- 退出登录

## 5.5 视频与媒体能力

如果产品主线是视频、音频或富媒体内容，这一层通常是技术投入最重的部分之一，主要围绕：

- Expo Video
- 媒体预加载
- 播放器池
- 屏幕方向管理
- 后台播放
- 画中画

再结合手势、全屏、滚动窗口和控件层设计，就可以形成一个接近内容平台级别的媒体基础设施。

## 5.6 UI 与交互层

常见 UI 技术体系包括：

- React Native Paper
- Reanimated
- Gesture Handler
- Expo Blur
- Expo Glass Effect
- Toast 系统
- Modal 系统
- 响应式尺寸工具

这一套组合适合支撑一套完整的主题化、可扩展设计系统和视觉效果系统。

## 5.7 表单与验证

表单层通常包括：

- React Hook Form
- Zod

这意味着认证、输入校验、密码设置等用户输入行为，都可以按类型安全和规则校验的方向组织。

## 5.8 数据库与本地数据层选型

对于需要离线能力、缓存或本地沉淀的移动项目，本地数据层常见选型包括：

- ORM / 查询构建层
- 迁移工具
- 本地数据库驱动

在重构项目里，这一层通常需要明确三件事：

- 本地数据层是否只是缓存
- 本地数据层是否承担离线真相来源
- 本地数据层与远程接口如何分工

## 6. 工程架构风格

内容型、交互较重的重构项目，通常都需要清晰的工程分层。

一种典型方式是 **Feature-Sliced Design (FSD)**，也就是把代码大体拆成：

- app
- pages
- widgets
- features
- entities
- shared

这意味着项目不是按“页面文件堆叠”的方式组织，而是尽量把：

- 应用入口
- 页面组合
- 用户交互能力
- 领域模型
- 通用基础设施

拆到不同层里。

对于重构项目来说，这类分层不只是目录规范，而是决定了内容系统、学习系统、认证系统和 UI 系统如何协作的核心工程原则。

## 7. 重构项目中推荐补充说明的内容

如果这份文档要被其他重构项目复用，建议在落地时额外补充以下说明，但不要把它们和产品定位混写在一起：

### 7.1 数据来源边界

- 哪些数据来自远程服务
- 哪些数据来自本地缓存或本地数据库
- 哪些数据仍是演示数据、历史数据或占位数据

### 7.2 业务闭环边界

- 哪些功能已经形成完整闭环
- 哪些功能只有页面和结构
- 哪些功能仍处于规划阶段

### 7.3 技术真相来源

- 哪些能力已接真实服务
- 哪些能力只是选型完成
- 哪些能力只是文档存在、尚未成为运行时真相

## 8. 推荐的理解方式

如果要给新接手重构项目的人一个最实用的理解框架，建议按下面顺序认知：

### 先看产品定位

先用一句话把产品讲清楚，例如“学习型短视频 App”“带沉淀空间的内容学习 App”“以内容流驱动的技能学习 App”。

### 再看核心主线

再梳理核心主线，例如：

- 登录
- 进入主入口
- 消费核心内容
- 借助学习辅助完成理解
- 将学习结果沉淀到个人空间

### 最后再补运行边界

如果需要写项目现状，应单独补“运行边界 / 数据边界 / 完成度边界”，不要混在总览定义里。

## 9. 总结

对于重构项目，一份高质量的总览文档应该做到三件事：

- 先讲清楚产品为什么存在，也就是整体需求
- 再讲清楚用户实际会看到什么，也就是 App 形态
- 最后讲清楚系统用什么搭起来，也就是技术栈与工程分层

只要把这三层拆开写，文档就不会既像产品介绍、又像实现清单、又像进度报告。这样整理出来的总览，不仅适合当前项目，也适合其他重构项目直接复用。
